#+TITLE: bookmark-plus — potential bugs (summary, no patches)
#+STARTUP: overview

* About this file

Short-form report of behaviors in =bookmark-plus= (upstream:
=dmgerman/bookmark-plus=, mirrored from EmacsWiki) that appear
to be bugs. Each entry describes the observed problem, how to
reproduce it, the root cause, and the proposed change in prose.

Patches with =diff= blocks live in the companion file
[[file:plus-potential-bugs.org][=plus-potential-bugs.org=]]. This
summary is intended as a first read: a way to decide which items
are worth pursuing before diving into any specific patch.

Environment: Emacs 30, modern =display-buffer= behavior,
=consult= and =vertico= as the completion UI, =pdf-tools=
installed.

Each entry ends with a link to the corresponding commit in
=dmgerman/bookmark-x=, where the downstream reference
implementation lives. Those commits are the *reference*, not
the patch — the companion file adapts them into diffs against
=bookmark-plus= file paths and function names.

* Bug 1 — =bookmark--jump-via= does not display for custom-handler jumps

** Symptom

Jumping to a bookmark whose record has a custom handler (PDF,
EWW, Gnus, Info, Man, image, org, or any third-party handler)
"works" — the destination buffer is created and its handler
runs — but no window changes. The user stays in the previously
selected buffer, unaware anything happened.

Related: when the caller sets =display-buffer-overriding-action=
(e.g. =bookmark-bmenu-switch-other-window=, previewing from
=*Bookmark List*=), some handler families additionally clobber
the originating window in place, so the "preview" replaces the
caller's buffer instead of opening in another window.

** Reproducer

1. Bookmark a PDF via =pdf-tools= (handler
   =pdf-view-bookmark-jump-handler=).
2. From any buffer, =M-x bookmark-jump= that bookmark.
3. Expected: PDF displayed in a window. Observed: nothing
   visible changes; the PDF buffer is alive but never shown.

For the preview variant, do the above from =*Bookmark List*=
via =C-o= and observe the =*Bookmark List*= window get replaced
by the handler's destination.

** Root cause

=bookmark-plus= redefines vanilla =bookmark--jump-via= (in
=bookmark+-1.el= at ~line 2916) so that it does *not* call
=DISPLAY-FUNCTION= on the destination after the handler
returns. The intent is to let the handler itself consult the
global =bmkp-jump-display-function= and decide what to do. Only
the default handler path (via =bmkp-goto-position=) actually
does so. Every custom handler follows the vanilla contract
("=set-buffer= to the destination and leave display to the
caller"), so nothing displays.

Handlers fall into three families that need distinguishing:

- *Case A* — =find-file= / =switch-to-buffer= inside the
  handler. These clobber the originating window's buffer in
  place, ignoring =display-buffer-overriding-action=. Hit by
  =org-bookmark-heading-jump= and any handler wrapping
  =find-file=.
- *Case B* — =pop-to-buffer-same-window= inside the handler,
  which honors the overriding action and displays the
  destination in another window itself. Hit by
  =org-goto-marker-or-bmk=.
- *Case C* — pure =set-buffer= handlers (e.g.
  =pdf-view-bookmark-jump-handler=). Caller is responsible for
  displaying.

An unconditional =funcall= of =DISPLAY-FUNCTION= after the
handler is correct only for Case C; Case A leaves the
originating window wrongly clobbered and then displays
elsewhere, and Case B produces a double display that steals a
second window (frequently =*Bookmark List*= in the preview
scenario) for the destination buffer.

** Two approaches to the fix

There are two ways to close the observed gap. Drew has a stated
design preference for the first; the downstream fork's original
patch implements the second. Both close the symptom; they
differ in how much of =bookmark--jump-via='s current contract
is preserved.

*** Approach A — per-handler fixes (Drew's preferred design)

Drew's position is that =bookmark--jump-via= should *not* be
changed to unconditionally call =DISPLAY-FUNCTION= after the
handler. The current behavior is deliberate: it lets a handler
choose to (1) not display at all, (2) invoke
=bmkp-jump-display-function= itself at the point of its
choosing, or (3) display in a way that ignores the caller's
requested action. A vanilla-style =funcall= at the end of
=bookmark--jump-via= removes that flexibility.

Under this approach, each handler that currently does only
=set-buffer= — and therefore silently fails to display under
=bookmark-plus= — is patched individually. At the end of the
handler body, the handler calls =bmkp-jump-display-function= on
the current buffer when that variable is bound and non-nil, and
raises the frame. This is the same small pattern already used
by Drew's fix for =xwidget-webkit-bookmark-jump-handler=.

Known handlers that need this treatment (non-exhaustive):

- =pdf-view-bookmark-jump-handler= (=pdf-tools=)
- =xwidget-webkit-bookmark-jump-handler= (vanilla Emacs)
- =org-bookmark-heading-jump= and other =find-file=-wrapping
  handlers (case A in the taxonomy above; here the handler
  should perform its own display rather than clobbering the
  current window)
- Any third-party handler that only =set-buffer='s

Trade-off: this preserves =bookmark-plus='s design intent and
avoids interfering with handlers that legitimately do their own
display. Coverage grows one handler at a time, however, and
=bookmark-plus= cannot rescue handlers whose upstream is
unwilling to patch.

*** Approach B — dispatch in =bookmark--jump-via=

Extend =bookmark--jump-via= to detect at run time which of the
three handler families actually ran (see Root cause) and
dispatch the display accordingly. The idea is to snapshot the
originating window's buffer before the handler runs, then
choose after: if the handler clobbered the origin, restore it
and call the display function; if the destination is already
showing somewhere, trust the handler; otherwise, call the
display function ourselves.

This covers every handler at once, including third-party ones
=bookmark-plus= cannot patch. The cost is that
=bookmark--jump-via='s contract moves closer to vanilla, which
conflicts with the minority of handlers that intentionally rely
on the current =bookmark-plus= behavior of not displaying, or
of displaying non-standardly.

*** Approach C — dispatch + warn (recommended)

Approaches A and B pull in opposite directions: A preserves
Drew's design (handlers own display) but leaves users hitting
the bug until every handler is patched upstream; B closes the
symptom for everyone but hides the fact that many handlers are
misbehaving.

Approach C combines them. Use Approach B's three-case dispatch
as a *safety net* so users never see the display bug. But in
the cases where the dispatch has to intervene — Case A (handler
clobbered the window, ignoring the caller's overriding action)
and Case C (handler followed the vanilla contract, leaving
display to the caller) — emit a =display-warning= naming the
handler and telling its maintainer exactly what to change.
Case B stays silent (that handler is a good citizen).

Effect: users get correct behavior today, and =bookmark-plus=
advocates for Drew's design at runtime — each affected handler
gets a specific, actionable message that its maintainer will
see in =*Warnings*=. Once handlers are fixed upstream, the
dispatch quietly falls through to Case B for them and the
warnings stop.

A =defcustom= (=bmkp-warn-on-handler-display-fix=, default =t=)
gates the warnings so power users can silence them if a
downstream handler is known-unfixable, while keeping the safety
net enabled.

*** Recommendation

Land Approach C. It gives users the safety net of Approach B
while advocating for Drew's design (Approach A) through the
warnings. Handler maintainers get a specific, actionable
message; users who don't care about that can flip
=bmkp-warn-on-handler-display-fix= to =nil= and keep the
safety net.

The smallest change common to all three approaches — replacing
=setq= with =let= for =bmkp-jump-display-function= inside
=bookmark--jump-via= — is orthogonal and worth landing on its
own; it keeps the dynamic binding scoped to the jump.

** Related: eager tooltip evaluation on auto-light

Caught in the same debugging pass, not the same bug.
=bmkp-make/move-overlay-of-style= sets =help-echo= to the eager
result of =bmkp-tooltip-content-function=. The default function
calls =image-dired-get-thumbnail-image= and =exiftool= for any
file matching =image-file-name-regexp=. With =pdf-tools=
installed and =pdf= in =image-file-name-extensions=, every
auto-light on a PDF bookmark spawns an ImageMagick PDF
conversion, which fails under default =policy.xml=
(CVE-2018-16509), leaking "Thumb could not be created ...
exited abnormally with code 1" errors.

Fix: pass a function to =help-echo= (Emacs accepts a function
there and calls it on demand when the user hovers), so the
tooltip content is computed lazily.

* Bug 2 — =bookmark-bmenu-switch-other-window= silently full-jumps

** Symptom

=C-o= in =*Bookmark List*=, documented as previewing the
bookmark "while the current window remains selected", instead
replaces the =*Bookmark List*= window's buffer with the
destination — a full jump.

** Reproducer

1. =M-x bookmark-bmenu-list=.
2. Move point to any bookmark.
3. =C-o= (=bookmark-bmenu-switch-other-window=).
4. Expected: destination shown in another window, =*Bookmark
   List*= still selected. Observed: =*Bookmark List*= replaced
   by destination in place.

** Root cause

=bookmark-plus= redefines =bookmark-bmenu-switch-other-window=
in =bookmark+-bmu.el= and let-binds =pop-up-windows=,
=same-window-buffer-names=, and =same-window-regexps= (obsolete
under modern =display-buffer=, Emacs 24+) before calling plain
=display-buffer= with no action. With the default action,
=display-buffer= is free to reuse the selected window for the
destination.

** Proposed change

Bind =display-buffer-overriding-action= with =(inhibit-same-
window . t)= and wrap in =save-selected-window= so focus
returns to =*Bookmark List*= even if the handler transiently
selects the destination.

Separately (not part of the bug fix, purely a usability tweak):
add =TAB= as an alias for =C-o= in =bookmark-bmenu-mode-map=,
matching the convention in =vertico=, =dired-preview=, and
similar list buffers.

** Origin

Downstream commit in =dmgerman/bookmark-x=:
[[https://github.com/dmgerman/bookmark-x/commit/be10f69][=be10f69=]].

* Bug 3 — =bmkp-jump-dired= ignores =display-buffer=

** Symptom

Previewing a Dired bookmark from =*Bookmark List*= (=C-o=)
replaces the =*Bookmark List*= window's buffer with the Dired
buffer, instead of showing Dired in another window. Only
affects records whose handler is =bmkp-jump-dired= (produced by
=bmkp-set= from inside a dired buffer). Plain file / autofile
bookmarks pointing at the same directory work correctly.

** Reproducer

1. =M-x dired /tmp/=.
2. =bmkp-set= to bookmark the dired buffer (handler
   =bmkp-jump-dired= is stored on the record).
3. =M-x bookmark-bmenu-list=, move to that bookmark, press
   =C-o=.
4. Expected: Dired opens in another window, =*Bookmark List*=
   remains selected. Observed: =*Bookmark List*= replaced by
   Dired in place.

This reproducer assumes Bug 2's fix is applied. Without it, the
=C-o= caller doesn't install =display-buffer-overriding-action=
in the first place, so the symptom looks the same but
originates from Bug 2, not Bug 3.

** Root cause

=bookmark-bmenu-switch-other-window= (with Bug 2 fixed)
installs =display-buffer-overriding-action= with =(inhibit-
same-window . t)= and dispatches via =display-buffer=. Inside
=bmkp-jump-dired=, the =cl-case= on
=bmkp-jump-display-function= folds =display-buffer= into the
same arm as the same-window defaults and calls plain =dired=,
which uses =switch-to-buffer= internally. =switch-to-buffer=
does not consult =display-buffer-overriding-action=, so it
replaces the =*Bookmark List*= buffer in its window.

Second bug hiding behind the first: the tail logic after the
=cl-case= calls =goto-char POS=. Without =set-buffer= in
effect, point moves in whichever buffer is current —
potentially =*Bookmark List*= — and the next read of the
bookmark at point returns the wrong record.

** Proposed change

Give =display-buffer= its own arm in the =cl-case=: build via
=dired-noselect=, display through =display-buffer= (which
honors the overriding action), and =set-buffer= so the tail
logic runs in the dired buffer.

** Note for the maintainer

The same shape of bug likely exists in =bmkp-jump-info=,
=bmkp-jump-eww=, =bmkp-jump-man=, and =bmkp-jump-w3m=. Not
audited; each calls a mode entry point that internally uses
=switch-to-buffer= or a display convention that ignores
=display-buffer-overriding-action=. Bug 1's Approach B would
paper over the symptoms for those handlers, but a per-handler
fix on the model of this patch is cleaner.

** Origin

Downstream commit in =dmgerman/bookmark-x=:
[[https://github.com/dmgerman/bookmark-x/commit/a07d516][=a07d516=]].

* Bug 5 — Save prompts to reload the file Emacs just wrote

** Status

Real gap between =bookmark-plus='s redefined
=bookmark-write-file= and vanilla behavior, but does not
produce the described symptom in a pure =bookmark-plus= session.
See below.

** Symptom (described, does not manifest in pure =bookmark-plus=)

If the user's bookmark file (=bookmark-default-file=, or
whichever file is tracked by =bookmark-bookmarks-timestamp=)
is the one being saved, one might expect Emacs to prompt on the
*next* access to reload it — because
=bookmark-maybe-load-default-file= would see the file's
modification time is newer than the recorded timestamp.

** Root cause

=bookmark-plus= redefines =bookmark-write-file= (in
=bookmark+-1.el= at ~line 3553). The redefined function writes
the file but does not update =bookmark-bookmarks-timestamp=.
Vanilla's =bookmark-write-file= updates it via
=bookmark-set-bookmarks-timestamp=; the =bookmark-plus=
redefinition dropped that call.

** Why the symptom does not appear

=bookmark-plus= also redefines =bookmark-maybe-load-default-
file= (=bookmark+-1.el= at ~line 2892) to use a boolean
=bmkp-bookmarks-already-loaded= instead of consulting
=bookmark-bookmarks-timestamp=. Since it never checks the
file's modification time, it never prompts to reload.

The reload-prompt symptom would only appear if vanilla
=bookmark-maybe-load-default-file= somehow ran while
=bookmark-plus= was loaded — an unusual configuration.

** Proposed change

If, for cross-tool compatibility, the maintainer wants to keep
=bookmark-bookmarks-timestamp= in sync anyway (so that vanilla
=bookmark= command interoperability doesn't misbehave), update
the timestamp at the end of =bookmark-write-file= when the
on-disk file is the one the built-in loader tracks. Skip the
ADD path — that flow writes records that are not in
=bookmark-alist=, and updating the timestamp there would hide
genuine external changes.

Otherwise, this entry can be dropped.

** Origin

Downstream commit in =dmgerman/bookmark-x=:
[[https://github.com/dmgerman/bookmark-x/commit/137f219][=137f219=]].

* Bug 6 — Consult-driven jump loads the file but does not display it

** Preamble

This entry is conditional: it presupposes a preview-aware
reader that routes =bookmark-jump= (or an equivalent
=bmkp-jump= variant) through =consult--read=. =bookmark-plus=
as shipped does not do that; the entry is included in case
such an integration lands.

** Symptom

With =consult= and =vertico= as the completion UI, invoking a
jump-through-consult path exhibits this sequence:

1. Preview shows the destination correctly during candidate
   navigation.
2. User selects a candidate (=RET=).
3. Consult clears the preview and returns; the destination file
   is loaded (buffer alive), but no window changes — the user
   is left in the calling buffer, not the destination.

** Root cause (working theory)

Consult's preview callback leaves the destination buffer alive
but undisplayed once the user selects. The subsequent
=bookmark-jump= path relies on =bookmark--jump-via= to call the
display function; for default-handler bookmarks that works, but
for custom-handler bookmarks whose handler only =set-buffer='s
(Bug 1 above), no display happens.

** Proposed change

*Display half.* Fixed by Bug 1 — once =bookmark--jump-via=
either dispatches display itself (Approach B) or every custom
handler learns to invoke =bmkp-jump-display-function= (Approach
A), the consult flow gets its display too. No additional
change needed here.

*Temporary-bookmark filter.* Already implemented in
=bookmark-plus=: =bookmark-write-file= (=bookmark+-1.el= at
~line 3604) wraps its record dump in
=(unless (bmkp-temporary-bookmark-p bmk) ...)=. Temporary
bookmarks do not survive across sessions. Nothing to add.

If a preview-aware consult wrapper is introduced later and the
display symptom persists after Bug 1's fix, revisit this entry
with a concrete reproducer.

** Origin

Downstream commit in =dmgerman/bookmark-x=:
[[https://github.com/dmgerman/bookmark-x/commit/9fb3420][=9fb3420=]].
The downstream fix is 150+ lines interleaved with the =bmkx-*=
