#+TITLE: bookmark-plus-gt
#+STARTUP: overview

Non-invasive extensions on top of [[https://github.com/dmgerman/bookmark-plus][Bookmark+]].  Symbols under =bmkp-gt-=.
Bookmark+ itself is not modified.

* Install

First, install bookmark-plus. You have to download the files from emacsWiki
https://www.emacswiki.org/emacs/BookmarkPlus
(do not use the github fork at https://github.com/emacswiki/bookmark-plus,
it is outdated). Place those files in its own directory and use package
to load them

#+begin_src emacs-lisp   :exports both
(use-package bookmark+
  :load-path "~/.emacs.d/modules/bookmark-plus"
  )
#+end_src

bookmark-plus has a lot of configurable features. Some of these are set by
bookmark-plus-gt in order it to properly work.

#+begin_src emacs-lisp
(use-package bookmark-plus-gt
  :load-path "~/.emacs.d/modules/bookmark-plus-gt"
  :after bookmark+
  :config
  ;; Every feature file is loaded automatically by requiring the
  ;; package.  Enable the ones you want with the corresponding mode:
  (bmkp-gt-bmenu-tags-mode 1)    ; tags and type columns in *Bookmark List*
  (bmkp-gt-auto-update-mode 1)   ; reading-position tracker
  (bmkp-gt-browsel-tabs-mode 1)) ; browser tabs as temporary URL bookmarks
#+end_src

Each optional feature is a global minor mode that is off by
default and attaches no side effects until enabled.  Toggle any
of them at any time; disabling removes the associated advice,
hooks, and keybindings.

Optional features:

- =Bookmark-plus-gt-jump= — the =consult=-backed jump reader
  (=bmkp-gt-jump= and its variants) plus =bmkp-gt-list-preview-mode=
  for =*Bookmark List*=.  See "Minibuffer filtering" and
  "Annotations" below.
  - Optional: =consult= (without it, the reader falls back to
    =bookmark-completing-read= and the narrow-key + live-preview
    features are unavailable), =marginalia= (adds the tag column and
    rewrites the type column in the annotation), =orderless= (needed
    for stacked =@Type ;tag= minibuffer filtering — see "Minibuffer
    filtering" below).
- =bmkp-gt-bmenu-tags-mode= — extra columns in =*Bookmark List*=
  between the name and the filename: a tags column (=;=) and a type
  column (=@=).  See "*Bookmark List* columns" below.
- =bmkp-gt-auto-update-mode= — the reading-position tracker:
  bookmarks carrying the =auto-update= property have their location
  refreshed as the file is read.  See "Auto-update bookmarks" below.
- =bmkp-gt-browsel-tabs-mode= — every open browser tab exposed as a
  temporary URL bookmark.  See "Browser tabs as temporary bookmarks"
  below.
  - Requires [[https://github.com/dmgerman/browsel][browsel]] version =0.94= or later (loaded and connected to
    at least one browser).  The =bookmark-plus-gt-browsel-tabs= file
    checks =browsel-version= at load time and signals an error if it
    is older.

Two things are always on when =bookmark-plus-gt= itself is loaded,
because they are not features you can meaningfully opt out of:

- The =bmkp-gt-relocate-here= and =bmkp-gt-relocate-this-file-here=
  commands (see "Commands" below).
- The =bookmark--jump-via= advice that ensures custom-handler jumps
  actually display their destination (a bookmark-plus bug fix).
- The =customize-save-variable= intercept that routes bookmark-plus
  runtime state (=bmkp-last-as-first-bookmark-file=) to
  =bmkp-gt-state-file= instead of =custom-file=.  See "State
  persistence" below.
- The =bookmark-bmenu-execute-deletions= advice that keeps point on
  a surviving bookmark after =x=.  See "Post-delete point" below.
- The safety guards around same-named bookmarks: force
  =bmkp-bookmark-set-confirms-overwrite-p= on, and warn if
  =bmkp-propertize-bookmark-names-flag= is off.  See "Same-named
  bookmarks" below.

* Commands

- =bmkp-gt-jump=, =bmkp-gt-jump-other-window=, =bmkp-gt-jump-other-frame= —
  drop-in replacements for =bookmark-jump= et al.  When =consult= is
  loaded (and =bmkp-gt-jump-use-consult-flag= is non-nil), the read
  is done through =consult--read= with live preview.  Falls back to
  =bookmark-completing-read= otherwise.  Bind them wherever you want;
  the standard Bookmark+ keys are left alone.

- =bmkp-gt-list-preview-mode= — buffer-local minor mode for
  =*Bookmark List*=.  With point on a row, the destination opens in
  another window (analog of =next-error-follow-minor-mode=).  For
  always-on:
  =(add-hook 'bookmark-bmenu-mode-hook #'bmkp-gt-list-preview-mode)=.

- =bmkp-gt-bmenu-toggle-tags= — show/hide the tags column in
  =*Bookmark List*=.  Bound to =;=.

- =bmkp-gt-bmenu-toggle-type= — show/hide the type column in
  =*Bookmark List*=.  Bound to =@=.

- =bmkp-gt-auto-update-toggle= / =bmkp-gt-auto-update-bmenu-toggle= —
  flip the =auto-update= property on a bookmark.  The bmenu variant
  reads the bookmark at point and is bound to =^= in =*Bookmark List*=.

- =bmkp-gt-auto-update-mode= — global minor mode; when on, refreshes
  every auto-update bookmark's position from the buffer visiting its
  file (idle timer + on kill-buffer + on window-state-change).  Off
  by default; enable with =(bmkp-gt-auto-update-mode 1)=.

- =bmkp-gt-auto-update-now= — force one refresh pass immediately.

- =bmkp-gt-browsel-tabs-refresh= — rebuild the browsel-tab bookmarks
  from live browser state.  Rarely needed by hand: the same refresh
  runs automatically before =bookmark-bmenu-list=,
  =bookmark-completing-read=, =bmkp-url-jump=, and
  =bmkp-url-jump-other-window=.

- =bmkp-gt-relocate-here= — point a bookmark (chosen by name) at the
  current buffer/point.  Preserves the name, tags, annotation,
  handler, and every other property; only the location fields
  (filename, buffer-name, position, context strings, end-position)
  are rewritten.  With a prefix argument (=C-u=), also sets the
  =auto-update= property on the bookmark so future ticks track it.
  Defined in the entry point and always available, independent of
  =bmkp-gt-features=.

- =bmkp-gt-relocate-this-file-here= — same as =bmkp-gt-relocate-here=,
  but scoped: only bookmarks pointing at the current file/buffer are
  offered.  If exactly one such bookmark exists it is used without
  prompting; the message reports which name was updated.  Errors if
  no bookmarks match.  Same =C-u= behavior.

* Minibuffer filtering (=bmkp-gt-jump=)

Every candidate carries hidden tokens that are part of the
candidate string but invisible in the display (via =display "")=.
Filtering by these tokens depends on the matching style
(=completion-styles=), not on the UI (vertico / ivy / helm /
default =completing-read= are all fine).

*Only =orderless= makes them fully useful.*  It splits input on
whitespace and matches each clause independently against the
raw candidate string, so =@Dired ;work= reduces the list to
Dired-category bookmarks tagged =work=.  Other built-in styles
handle the tokens poorly: =substring= matches only a single
contiguous input (so =@Dired= alone works, =@Dired ;work= does
not); =basic= and =partial-completion= only prefix-match and see
nothing; =flex= is too permissive to be a useful filter.

- =@Type= — group label from =bmkp-gt-jump-narrow=
  (=@File=, =@Dired=, =@Web=, =@Magit=, =@Epub=, …).  =@= avoids
  Orderless's =!= negation dispatcher.
- =;tag= — every tag on the bookmark.  Stackable (=;work ;urgent=).

Same characters appear as toggles in =*Bookmark List*= (=;= toggles
the tags column, =@= toggles the type column), so the mapping
=;=→tags, =@=→type is consistent across both interfaces.

Long-name display.  Names longer than
=bmkp-gt-jump-name-max-width= (default =50=) render in the
minibuffer as =<head>…<tail>=, where <tail> is
=bmkp-gt-jump-name-tail-keep= characters (default =10=).  The
truncation shortens the candidate STRING (not just its display),
so consult and marginalia size the annotation column from the
truncated length — a single long candidate no longer widens the
whole list.  The elided middle is not matchable by orderless /
substring / flex; only the visible head and tail are.  The full
untouched name is still recoverable from the
=bmkp-gt-bookmark-name= text property attached to each candidate,
so bookmark lookup, jump, and annotation display all resolve
against the real name.  Set the cap to =nil= to disable
truncation.

Facet keys inside the minibuffer:

- =M-t= — add a tag filter (completing-read from all tags).
- =M-D= — pop the last filter.
- =M-T= — clear all filters.

** Type narrowing (single-key filter)

Consult's narrow mechanism gives a single-key alias for each type
group.  The prefix is =consult-narrow-key= (default =<=; set it to
=C-== or similar if you prefer a modifier).  Then in the
=bmkp-gt-jump= minibuffer:

- =PREFIX f= — File
- =PREFIX d= — Dired
- =PREFIX i= — Info
- =PREFIX o= — Org heading
- =PREFIX n= — News/Gnus
- =PREFIX m= — Man/Help
- =PREFIX p= — Picture
- =PREFIX w= — Web
- =PREFIX u= — URL
- =PREFIX s= — Shell
- =PREFIX v= — Doc view / PDF
- =PREFIX D= — Desktop
- =PREFIX S= — Snippet
- =PREFIX g= — Magit
- =PREFIX b= — Browser tab (added when =bookmark-plus-gt-browsel-tabs= is loaded)
- =PREFIX ?= — help menu listing every char with its label
- =PREFIX SPC= (or press the prefix again with no char) — clear
  narrowing

The full mapping (label, char, matching handlers) is
=bmkp-gt-jump-narrow=; edit it to rename, add, or remove groups.
The narrowing is filtered by consult *before* orderless matches
the input string, so a narrow + typed query composes as an AND.

Without consult, the same taxonomy is available via the =@Type=
hidden token above (e.g. type =@Dired= in the minibuffer).  Both
mechanisms are derived from the same =bmkp-gt-jump-narrow= list.

** Sorting and grouping candidates

Two defcustoms control candidate order in the =bmkp-gt-jump= reader:

- =bmkp-gt-jump-sort-by= (default =mru=):
  - =mru= — most recently visited first, from each bookmark's
    =last-visited= property (updated by Bookmark+'s
    =bmkp-record-visit= on every jump).
  - =visits= — highest =visits= counter first.
  - =alpha= — alphabetical by bookmark name.

- =bmkp-gt-jump-group-by= (default =nil=):
  - =nil= — no grouping.
  - =type= — group by handler-type (=@File=, =@Dired=, …).
  - =tag= — group by primary tag.

  When grouping is on, the sort applies within each group; groups
  themselves appear in first-seen order, so the group holding the
  best-sorting candidate leads.

* =*Bookmark List*= columns

With =bookmark-plus-gt-tags= loaded, the list buffer renders two
extra columns between the name and the filename:

: [marks] [name] [;tag ;tag …] [Type] [filename]

Both columns are on by default and toggleable:

- =;= — =bmkp-gt-bmenu-toggle-tags=
- =@= — =bmkp-gt-bmenu-toggle-type=

The type label comes from =bmkp-gt-jump-narrow= (same source as the
consult minibuffer), so =@File=, =@Dired=, =@Web=, etc. mean the same
thing everywhere.

Face continuity.  The tag and type text is propertized with the
row's own row-type =font-lock-face= (=bmkp-url=,
=bmkp-local-file-without-region=, =bmkp-info=, ...) — the same
face upstream applies to the whole =[marks, max-width]= range.
That means any background from the row-type face flows
continuously across name + padding + tag + type, so faces with a
custom background do not stop mid-row and reveal the padding as a
seam.

Column widths.  The tags and type columns are capped by
=bmkp-gt-bmenu-tags-max-width= (40) and =bmkp-gt-bmenu-type-max-width=
(20).  The name column has its own cap,
=bmkp-gt-bmenu-name-max-width= (default =50=): the tags/type start
columns are computed from =min(widest-name, cap)=, and each row's
name is smart-truncated in the buffer at render time so no row
extends past the cap and every column aligns.  Truncation keeps
the first =cap - tail-keep - 1= characters, inserts a single
ellipsis (=…=), and keeps the last =bmkp-gt-bmenu-name-tail-keep=
characters (default =10=).  The buffer text is rewritten but the
row's =bmkp-full-record= and =bmkp-bookmark-name= text properties
are preserved onto the truncated span, so =bookmark-bmenu-bookmark=
and every other row lookup still returns the full name — only the
rendered name is shortened.  Set the cap to =nil= to disable
truncation and align on the true widest name (the pre-cap
behavior).  The cap has no effect when both tags and type columns
are hidden — the full name renders naturally.

* Auto-update bookmarks

A bookmark carrying the =auto-update= property is a *reading-position
tracker*.  While =bmkp-gt-auto-update-mode= is on, three triggers
refresh the bookmark's =position= (and context strings) to the
current point of the buffer visiting its file:

- an idle timer firing every =bmkp-gt-auto-update-interval= seconds;
- =kill-buffer-hook= (catches closing the file);
- =window-state-change-hook= (catches switching away).

Bookmarks without the property are not touched.  Files whose buffer
is not currently visited are not touched either.  =end-position= is
pinned to =position= on each refresh so upstream's region-restore
path (which errors on region-less bookmarks under =bmkp-use-region=)
is never taken for an auto-update bookmark.

The refresh uses the buffer's own =bookmark-make-record-function=,
so it captures the fields the buffer's mode considers meaningful —
=position= and context strings for text, =page= / =slice= / =size=
/ =origin= for PDF, =info-node= for Info, =location= for EWW,
=dired-directory= et al. for Dired, and so on.  Any mode that ships
a proper record function is handled automatically; no per-mode list
to maintain.  Identity/history fields (=id=, =tags=, =annotation=,
=auto-update=, =created=, =visits=, =last-visited=, =defaults=) are
always preserved.

Display: =*Bookmark List*= renders a fifth marks column with =^= on
each auto-update bookmark, using face =bmkp-gt-auto-update-mark= while
the mode is on and =bmkp-gt-auto-update-mark-inactive= (inverse-video)
while it is off — so a set property with no active tracking is
visually flagged.

Keys and commands: =^= in =*Bookmark List*= toggles the property on
the bookmark at point (see =bmkp-gt-auto-update-bmenu-toggle=); from
anywhere, =M-x bmkp-gt-auto-update-toggle=.  =M-x
bmkp-gt-auto-update-mode= turns tracking on/off globally; =M-x
bmkp-gt-auto-update-now= forces one refresh pass.  Press =?= in the
list buffer to see the =^= entry in the marks legend, including a
live status of whether the mode is on or off.

The toggle does not itself call =bookmark-save= (matches
=bmkp-toggle-temporary-bookmark=); the property change is tracked in
=bmkp-modified-bookmarks= and persists on the next Bookmark+ save.

=C-h RET= (=bmkp-describe-bookmark=) on an auto-update bookmark shows
an =Auto-update: yes= line in the description, immediately above
=Tags:=.  Bookmarks without the property render exactly as upstream.

* Browser tabs as temporary bookmarks

With =bookmark-plus-gt-browsel-tabs= loaded, every tab open in the
connected browsers (via [[https://github.com/dmgerman/browsel][browsel]]) is exposed as a temporary URL
bookmark.  The bookmark name is the tab URL — suffixed =<2>=,
=<3>=, ... on collision — and the tab title is the annotation.
Records carry the =bmkp-temp= property, so they are never written
to =bookmark-default-file=.  A single tag equal to the browsel
client name (=chrome=, =firefox=, ...) is attached to each record;
the tag renders as =;chrome= / =;firefox= in the =*Bookmark List*=
tags column and is filterable via bookmark-plus's tag machinery.

Recognition and jump:

- =bmkp-url-bookmark-p= is advised so browsel-tab bookmarks count
  as URL bookmarks.  URL-only commands like =bmkp-url-jump= and
  filters like =bmkp-bmenu-show-only-urls= pick them up
  automatically.
- A new entry is added to =bmkp-gt-jump-narrow= (when the preview
  module is loaded): =(?b "Browser tab" bmkp-gt-browsel-tabs-jump)=.
  So the type column in =*Bookmark List*= shows =Browser tab= for
  these records, and the consult narrow key =b= (with the
  =consult-narrow-key= prefix) filters =bmkp-gt-jump= to browsel
  tabs.  Same for the minibuffer facet =@Browser tab=.
- Jumping calls =browsel-focus-tab= with the tab's recorded id and
  browser, and with =FOCUS-WINDOW= set so the browser's OS window
  is raised.  If the tab has since closed or the browser is no
  longer connected, the handler falls back to =browsel-browse-url=
  with the recorded URL.
- After focusing, the handler throws to =bookmark--jump-via= so
  the standard post-handler chain (annotation display, auto-light,
  fringe mark) does not run and steal focus back from the browser.

Auto-refresh: the tab set is rebuilt from live browser state
before any of the following functions runs, so you never see a
stale list:

- =bookmark-bmenu-list=
- =bookmark-completing-read=
- =bmkp-url-jump=
- =bmkp-url-jump-other-window=

=M-x bmkp-gt-browsel-tabs-refresh= forces a rebuild on demand.

Browser selection: =bmkp-gt-browsel-tabs-browsers= is passed
unchanged to =browsel-browser-tabs=.

- =nil= (default) — poll every connected browser.
- A string — that single browser only.
- A list of strings — every browser in the list.

Filtering: =bmkp-gt-browsel-tabs-filter= is applied per tab
before it becomes a bookmark.

- =nil= (default) — include every tab.
- Regexp string — include a tab only if its URL matches.
- Function — called with the tab plist; non-nil to include.

Disconnection: if fetching the tab list fails (no browser
connected, request timeout, etc.) a warning is issued and the
browsel-tab set is left empty.  All other bookmarks keep working
untouched; the next successful refresh re-populates.

Delete-closes-tab: deleting a browsel-tab bookmark closes the
underlying browser tab.  =bookmark-delete= is advised =:before= to
call =browsel-close-tab= with the record's =browsel-id= and
=browsel-browser= when the record is a browsel-tab bookmark.
Covers both =M-x bookmark-delete= and the =*Bookmark List*=
mark-and-execute flow (=d= then =x=).  Errors from the close call
(tab already gone, browser disconnected) are demoted to a warning
so the bookmark-side deletion still proceeds.  Non-browsel-tab
bookmarks are untouched by this advice.

* Annotations

When marginalia is loaded, =bmkp-gt-bookmark-annotation= registers on
the =bookmark= category and adds a padded =;tag= column, then
rewrites marginalia's type column to use =bmkp-gt-jump-narrow=
labels (so what you see matches what =@= filters).  Non-file
bookmarks show their =location= URL instead of the =- no file -=
placeholder.

* State persistence

Bookmark+ persists a couple of internal state variables (currently
=bmkp-last-as-first-bookmark-file=) via =customize-save-variable=,
which routes through Emacs's =custom-file=.  Users with
=custom-file= set to =/dev/null= (or any unwritable path) as a
"config-is-code, no Customize crumbs" gesture hit a =file-error=
every time upstream tries to persist, which aborts the enclosing
=bookmark-save= and cascades into visible bugs such as the bookmark
list failing to refresh after =T +=.

Bookmark-plus-gt intercepts =customize-save-variable= for a fixed
allowlist of symbols and routes their persistence to a dedicated
file, =bmkp-gt-state-file= (default
=~/.emacs.d/bookmark-plus-gt-state.el=).  All other symbols pass
through to Emacs's normal Customize machinery unchanged.

The allowlist is =bmkp-gt--persisted-vars= — currently just
=bmkp-last-as-first-bookmark-file=.  Extend by =add-to-list=.  The
state file is auto-generated (a sequence of =setq= forms); safe to
delete — it will be rewritten on next persist.

* Post-delete point

Upstream's =bookmark-bmenu-execute-deletions= (bound to =x=) does
not reliably restore point after the buffer is rebuilt.  In the
common case where the user marks the row they are on and presses
=x=, upstream's fallback lands point in the header area or on the
first bookmark rather than on the row after the one that was
deleted.

Bookmark-plus-gt advises the command =:around=.  Before calling
upstream it records the nearest surviving bookmark — the next
unmarked row below point, falling back to the previous unmarked
row above — and after upstream returns it navigates to that row.
Both the pre- and post-passes are wrapped in =condition-case=, so
a failure inside the advice can never block the underlying
deletion; it is logged as a warning and upstream still runs.

* Same-named bookmarks

Bookmark+ can hold several bookmarks with the same literal name.

*To create a new bookmark whose name matches an existing one*, use
=C-u M-x bookmark-set= (Lisp equivalent: =(bookmark-set NAME '(4) nil)=).
Without the prefix argument, =bookmark-set= does NOT create a new
bookmark — it overwrites the existing record in place and copies
that record's =tags= and =annotation= onto the update (see "Merge
risk on =bookmark-set=" below).

Two paths to get there:

- =C-u M-x bookmark-set= at a new location with the same name.
  With =bmkp-propertize-bookmark-names-flag= non-nil (the default on
  Emacs > 20), each bookmark's name string carries a
  =bmkp-full-record= text property that identifies its specific
  record, so both entries coexist in =bookmark-alist= and every
  bookmark-plus command that takes a name (jump, delete, rename,
  add-tags, edit) resolves the intended one unambiguously.
- =M-x bookmark-load FILE= where FILE contains a name that already
  exists in the current session.  Upstream renames the *incoming*
  bookmark to =NAME<2>=, =NAME<3>= ... on load; both survive as
  distinct alist entries.  =C-u M-x bookmark-load= switches
  bookmark files instead — the current list is replaced entirely,
  so there is no collision case.

Merge risk on =bookmark-set=.  Plain (no prefix) =M-x bookmark-set
NAME= at a *different* location, when NAME already exists,
silently *merges* into the existing bookmark: the new record picks
up the old bookmark's =tags= and =annotation= (per
=bmkp-properties-to-keep=, whose default is =(tags annotation)=)
and then location / position / filename are overwritten to the
current point.  Result: one bookmark with the old tags and the new
location — not two bookmarks.

Bookmark-plus-gt forces =bmkp-bookmark-set-confirms-overwrite-p= to
=t= at load time so =bookmark-set= now prompts =Overwrite bookmark
'NAME'? (y/n)= on collision.  Answer =n= to abort and rename
manually; answer =y= to accept upstream's merge behavior.

Bookmark-plus-gt also verifies at load time that
=bmkp-propertize-bookmark-names-flag= is non-nil.  If it is nil, a
=display-warning= is issued: without it, same-named bookmarks
cannot be distinguished reliably by bookmark-plus commands.  The
flag is not force-set (the user may have disabled it deliberately).

Editing safety.  =<2>=, =<3>=, ... suffixed names are literal,
distinct names in =bookmark-alist=.  Bookmark-plus resolves each by
its full name, so editing =test= never touches =test<2>=.  Same for
tags: =bmkp-add-tags "test"= adds only to =test=, not to
=test<2>=.

Display order.  With the default alphabetical =bmkp-sort-comparer=,
=NAME= sorts before =NAME<2>=, =NAME<3>=, ..., =NAME<9>= (=NAME= is
a proper prefix, and the single-digit suffixes compare numerically
via string ordering).  Ten or more collisions would sort =NAME<10>=
between =NAME<1>= and =NAME<2>= — unlikely in practice.  Other
sort orders (=mru=, =visits=, ...) sort independently of the
suffix.

** Load-order sort

Bookmark files form a *stack*: the file loaded most recently is
on top, previously-loaded files are below.  Bookmark-plus does
not track load order out of the box.  Bookmark-plus-gt tags every
loaded bookmark with a `bmkp-gt-load-index' property and provides
the primary comparer =bmkp-gt-sort-by-load-order= plus the
secondary comparer =bmkp-gt-creation-oldest-cp= (ascending by the
upstream `created' timestamp).  The user chains them via
=bmkp-sort-comparer='s list-of-predicates form:

#+begin_src emacs-lisp
(setq bmkp-sort-comparer
      '((bmkp-gt-sort-by-load-order bmkp-gt-creation-oldest-cp)
        bmkp-alpha-p))
#+end_src

- =bmkp-gt-sort-by-load-order= (first predicate).  The
  most-recently-loaded file's bookmarks sort first, then the
  next-most-recent, and so on.  The initial auto-loaded bookmark
  file gets index =1= (the auto-load path goes through
  =bookmark-load= and picks up our advice); a subsequent =M-x
  bookmark-load= gets index =2=; and so on.  Bookmarks created
  in-session via =bookmark-set= carry no index — the comparer
  treats them as =most-positive-fixnum= so they sort at the top
  with each other.  Ties (same load-index) fall through to the
  next predicate.
- =bmkp-gt-creation-oldest-cp= (second predicate).  Within a
  single load, the bookmark created earlier sorts first
  (ascending by the =created= timestamp).  Bookmarks that lack
  the timestamp — created by an older bookmark-plus version, or
  the entry has been stripped — return nil and fall through.
- =bmkp-alpha-p= (final predicate).  Standard alphabetical tie-
  breaker; catches any pair that neither of the above resolved.

Deterministic suffix numbering.  Bookmark-plus's `bookmark-load'
assigns suffixes deterministically: for each incoming name that
collides with an existing bookmark, it picks the smallest integer
=n= such that =NAME<n>= is not already taken.  Suffixes are
applied in the order the incoming file's bookmarks are visited.
Same session, same load order → same suffixes; suffixes are
persisted verbatim when bookmarks are saved, so subsequent loads
see them as the literal names.

* File rename tracking

When a file or directory is renamed through Emacs — =dired= =R=,
=rename-visited-file=, or any Lisp call to =rename-file= — bookmarks
whose target is the renamed path are rewritten to point at the new
path.  Directory renames also carry over: any bookmark whose target
lives *under* the renamed directory has its path rewritten with the
new directory prefix.

Non-file bookmarks (URL, EWW, and anything else whose filename slot
holds =bmkp-non-file-filename=) are skipped.  Renames performed
*outside* Emacs — shell =mv=, Finder, =git mv= — are not tracked;
file-notify watches would be needed for that and are much heavier.

Implemented as always-on =:after= advice on =rename-file=
(=bmkp-gt-on-file-rename=).  The OS-level rename has already
succeeded by the time the advice runs, so it never signals: two
=condition-case= traps (one per bookmark, one outer) demote any
error to a =message= so a successful rename cannot appear to fail.

On success a summary is echoed:
=bmkp-gt: updated N bookmark(s) after rename of PATH=.
=bookmark-alist-modification-count= is incremented so the change is
picked up on the next save.

* Preview errors

Preview attempts (both paths) run under =condition-case=.  Failure
prints one line —
=bmkp-gt: preview failed for NAME: ERROR= — rate-limited to one per
bookmark name.
